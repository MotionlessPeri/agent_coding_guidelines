# Blueprint 程序化创建 Override：`AddFunctionGraph` 的模板参数陷阱

## 核心规则

程序化创建"父类函数的 Override"时，`FBlueprintEditorUtils::AddFunctionGraph` 的模板参数**必须是 `UClass*`**，不是 `UFunction*`。

```cpp
// 正确 —— override 路径
UFunction* OutFunc = nullptr;
UClass* OverrideClass = FBlueprintEditorUtils::GetOverrideFunctionClass(BP, FuncName, &OutFunc);

UEdGraph* Graph = FBlueprintEditorUtils::CreateNewGraph(
    BP, FuncName, UEdGraph::StaticClass(), UEdGraphSchema_K2::StaticClass());

FBlueprintEditorUtils::AddFunctionGraph<UClass>(
    BP, Graph, /*bIsUserCreated=*/false, OverrideClass);
```

```cpp
// 错误 —— 走"新函数"路径，导致函数同名冲突 + 签名不匹配
FBlueprintEditorUtils::AddFunctionGraph<UFunction>(
    BP, Graph, /*bIsUserCreated=*/false, ParentUFunction);  // ❌
```

## 为什么模板参数决定一切

`AddFunctionGraph` 内部调 `CreateFunctionGraphTerminators`，这个函数在 `UEdGraphSchema_K2` 里**重载了两个版本**（`EdGraphSchema_K2.cpp`）：

| 重载 | 参数类型 | 语义 | Entry 节点的 `FunctionReference.MemberParent` |
|------|---------|------|----------------------------------------------|
| A | `UClass*` | override 路径 | 父类指针（`UConditionEvaluator`） |
| B | `const UFunction*` | 新函数路径 | `nullptr`（=self） |

**编译器靠 `MemberParent` 识别 override**：
- 重载 A：Entry 节点挂到父类，pin/返回值**自动从父类 UFunction 推导**，`FUNC_Const/Static/BlueprintPure` flag 拷自父类
- 重载 B：Entry 节点 `MemberParent=nullptr`，调 `CreateUserDefinedPinsForFunctionEntryExit` 把签名**复制一份**作为 user-defined pin → 编译器视作"本 BP 声明的新函数"

## 错误路径的典型报错

传错成 `UFunction*` 时，编译 BP 报三条：

```
The function name in node XXX is already used
Cannot order parameters ReturnValue in function XXX
Cannot override '::XXX' at XXX which was declared in a parent with a different signature
```

对应三个后果：
1. 父类已有 `XXX`，子类又"声明"同名新函数 → 名字冲突
2. User-defined pin 复制时把 `ReturnValue` 自动改名成 `Return Value 1`（避免和"继承来的"重名）→ 参数顺序校验失败
3. `FUNC_Const` 等 flag 没从父类拷过来 → vtable 签名不匹配

## `bIsUserCreated` 是什么

**与走哪条路径正交的另一个开关**。控制 Entry 节点是否额外打 `FUNC_BlueprintCallable | FUNC_BlueprintEvent | FUNC_Public` flag。

来自 `BlueprintEditorUtils.h` 的 `CreateFunctionGraph` 模板实现：

```cpp
Schema->CreateFunctionGraphTerminators(*Graph, SignatureFromObject);  // 选重载

if (bIsUserCreated)
{
    int32 ExtraFunctionFlags = (FUNC_BlueprintCallable | FUNC_BlueprintEvent | FUNC_Public);
    // 补上 flag
}
```

| 场景 | `bIsUserCreated` | 理由 |
|------|------|------|
| 用户从 My Blueprint "+" 新建函数 | `true` | 完全新创，需要显式标 Callable/Event/Public |
| Override 父类函数 | `false` | 继承父类的可见性和调用约定 |
| `ImplementNewInterface` 实现接口 | `false` | 同上 |

**两个参数组合的实际含义**：

| 模板参数 | bIsUserCreated | 含义 |
|---------|---------------|------|
| `UClass*` | `false` | **Override（标准用法）** |
| `UFunction*` | `true` | 新建函数（签名取自 UFunction） |
| `UFunction*` | `false` | ❌ 既不是 override 也不是可调用新函数（四不像） |
| `UClass*` | `true` | 罕见（override 还手动加 flag） |

## 权威参考

UE 自己的 "Override Function" 菜单（My Blueprint 面板）就是这么做的：

```cpp
// Engine/Source/Editor/Kismet/Private/SMyBlueprint.cpp:2814 (UE 5.5)
void SMyBlueprint::ImplementFunction(...)
{
    UFunction* OverrideFunc = nullptr;
    UClass* const OverrideFuncClass = FBlueprintEditorUtils::GetOverrideFunctionClass(
        BlueprintObj, GraphAction->FuncName, &OverrideFunc);
    check(OverrideFunc);

    UEdGraph* const NewGraph = FBlueprintEditorUtils::CreateNewGraph(
        BlueprintObj, GraphAction->FuncName,
        UEdGraph::StaticClass(), UEdGraphSchema_K2::StaticClass());

    FBlueprintEditorUtils::AddFunctionGraph(
        BlueprintObj, NewGraph, /*bIsUserCreated=*/false, OverrideFuncClass);
}
```

做程序化 override 时对着抄，不要自创组合。

## Event vs Function 分流

父类带 `BlueprintImplementableEvent` 的方法，在 BP 子类里的 Override 形态**取决于返回值**：

| 返回值 | 形态 | 用什么 API 创建 |
|--------|------|----------------|
| `void` | EventGraph 里的 **Event 节点** | `FKismetEditorUtilities::RegisterAutoGeneratedDefaultEvent` 或 `AddDefaultEventNode` |
| 非 void | **Function Override**（独立 Function Graph） | 本文档描述的 `AddFunctionGraph<UClass>` 路径 |

判断依据是 `UEdGraphSchema_K2::FunctionCanBePlacedAsEvent(Function)` —— 返回 false 就只能走 Function Override。

## 一句话记忆

**模板参数选策略，`bIsUserCreated` 选挂牌**。选错任何一个都会造出一个既不能 override 又不能调用的"半成品"函数，编译时必崩。
